# 调研文档模板

## 通用文档头

每篇文档必须以此开头：

```markdown
# 标题

> 更新时间：YYYY-MM-DD
> 数据来源：列出主要参考来源（官方文档、博客文章、研究报告等）

---
```

---

## 模板 B: 深度调研

```markdown
# [技术名称] 深度调研

> 更新时间 / 数据来源

---

## 1. 概述与背景
- 是什么、解决什么问题、诞生背景

## 2. 核心架构与原理
- 整体架构图（mermaid 或文本格式）
- 关键组件说明
- 数据流 / 工作原理

## 3. 关键特性详解
- 每个特性：是什么、怎么用、适用场景

## 4. 性能与基准测试
- 官方 / 第三方 benchmark 数据
- 性能特点与瓶颈

## 5. 部署与运维
- 部署方式、运维要求、监控方案

## 6. 生态与社区
- 社区活跃度、周边工具、学习资源

## 7. 优劣势总结
| 优势 | 劣势 |
|------|------|

## 8. 适用场景与建议
- 推荐使用的场景
- 不推荐的场景
- 注意事项
```

## 模板 C: 选型评估

```markdown
# [技术/方案] 选型评估

> 更新时间 / 数据来源

---

## 1. 需求背景与评估目标
- 业务场景描述
- 核心需求列表
- 评估目标

## 2. 候选方案概览
| 方案 | 一句话描述 | 适用定位 |
|------|-----------|----------|

## 3. 评分标准
| 分值 | 含义 | 判定示例 |
|------|------|----------|
| 1 | 完全不满足 | 关键能力缺失 |
| 2 | 基本不满足 | 需大量定制 |
| 3 | 基本满足 | 满足下限要求 |
| 4 | 较好满足 | 超出基础需求 |
| 5 | 完全契合 | 原生支持核心场景 |

> 加权总分 = Σ(维度评分 × 维度权重)，权重之和必须为 1.0。所有方案使用同一份评分尺度。

## 4. 评估维度与权重
| 维度 | 权重 | 说明 |
|------|------|------|

## 5. 逐项评估
### 5.1 [方案A]
每个维度的评估 + 评分（1-5）+ 简短理由

### 5.2 [方案B]
...

## 6. 评估汇总
| 维度 | 权重 | 方案A | 方案B | ... |
|------|------|-------|-------|-----|
| ... |  |  |  |  |
| **加权总分** | 1.0 |  |  |  |

## 7. 风险分析
| 方案 | 主要风险 | 缓解措施 |
|------|----------|----------|

## 8. 结论与推荐
- 推荐方案及理由（结合加权总分与风险分析）
- 实施建议
- 后续跟进事项
```

## 模板 D: 全景扫描

```markdown
# [问题域] 技术全景扫描

> 更新时间 / 数据来源

---

## 1. 问题域描述
- 要解决的核心问题
- 关键需求特征

## 2. 方案分类概览
| 分类 | 代表方案 | 核心思路 |
|------|----------|----------|

## 3. 各方案简介
### 3.1 [方案名]
（每个方案 100-200 字简介：是什么、核心特点、适用场景）

## 4. 特性对比矩阵
| 特性 | 方案A | 方案B | 方案C | ... |
|------|-------|-------|-------|-----|

## 5. 成熟度与生态矩阵
| 方案 | 诞生年份 | GitHub Stars | 最近 release | 主要采用方 | 成熟度等级 |
|------|----------|--------------|--------------|------------|------------|

> 成熟度等级：**实验性**（早期 PoC、API 不稳定）/ **早期采用**（少量生产用户、文档不全）/ **主流稳定**（多家头部公司在用、API 稳定）/ **行业标准**（事实标准、生态完整）。

## 6. 推荐路径
按不同场景给出推荐选择路径：
- 如果你需要 X → 推荐 A
- 如果你需要 Y → 推荐 B

> 推荐时应同时考虑特性匹配度与成熟度等级，避免把实验性方案推荐给生产场景。
```
