# 信息源指引

调研过程中信息收集的来源优先级、搜索顺序、停止条件与时效性阈值。

## 一、来源优先级

**一级来源（最高优先级）**：
- 官方文档、技术规范、API 参考
- 项目维护方发布的白皮书 / RFC

**二级来源**：
- GitHub 仓库（README、Stars、commits、issues、discussions）
- 行业标准文档（OWASP、NIST、ISO、IETF RFC 等）

**三级来源**：
- 技术博客、行业分析报告、benchmark 报告
- 安全公告、CVE 记录
- 知名会议演讲（QCon、KubeCon、Strata 等）

**四级来源（辅助验证）**：
- 社区讨论（Stack Overflow、Reddit、知乎、V2EX）
- GitHub Issues / Discussions 中的用户实践
- 非官方教程、个人技术博客

> 关键结论必须以 ≥ 2 个来源交叉验证，且至少包含 1 个一级或二级来源。快速模式可放宽至 1 个一级来源。

## 二、推荐搜索顺序

按以下顺序逐步检索，先建立基础事实再补全细节：

1. **官方文档** → 锚定基础事实（架构、API、官方推荐场景）
2. **GitHub 仓库（README / Issues / Discussions）** → 了解真实使用问题与活跃度
3. **官方 / 独立 benchmark** → 性能数据与瓶颈
4. **行业分析文章（< 1 年）** → 趋势、采用情况、对比观点
5. **社区讨论** → 边界场景、踩坑案例

搜索查询构造原则：
- 用当前年份限定搜索（例如 `Kafka KRaft 2026`），确保时效性
- 中英文关键词都搜，扩大覆盖面
- 同义词扩展：`迁移` / `migration`、`性能` / `performance` / `benchmark`

## 三、信息饱和停止条件

满足以下任一条件即可停止当前维度的信息收集，避免陷入无限搜索：

- 当前维度已有 ≥ 2 个独立来源结论一致
- 新增搜索连续 3 次未带来新信息（仅重复已有结论）
- 已覆盖该维度的所有官方文档章节
- 快速模式下，已获取 1 个一级来源 + 1 个补充验证

> 出现矛盾信息时不能停止收集，必须追溯到根源或标注分歧。

## 四、时效性阈值

按技术演进速度分类，超过阈值需在文档中标注"可能过时"：

| 技术类型 | 数据过期阈值 | 示例 |
|----------|--------------|------|
| **快速迭代**：AI / LLM 框架、前端框架、云原生 | > 12 个月 | LangChain、Next.js、Kubernetes、新发布 ≤ 2 年的开源项目 |
| **中速演进**：中间件、数据库新版本、语言新版本 | > 24 个月 | Kafka、Redis、Python、TiDB |
| **稳定技术**：成熟数据库、操作系统、协议 | > 36 个月 | MySQL InnoDB、Linux 内核、HTTP/2、PostgreSQL |

判定规则：
- 不确定归类时按"中速演进"处理
- 同一技术不同模块演进速度不同时，取最快的那个分类
- 标注格式：`> 数据来源：XX 官方文档（2023-05，可能过时）`

## 五、矛盾信息处理

发现来源间结论矛盾时按以下流程处理：

1. 标注分歧：在文档中明确列出对立观点及各自来源
2. 追溯原因：版本差异？测试条件差异？立场差异？
3. 给出判断：若有充分证据，给出倾向性结论并说明依据；若无，明确标注"信息有限"
4. 关联信心评级：矛盾未解决的结论信心等级不应高于"中"
