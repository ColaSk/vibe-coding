## 1. 首先要选对产品线

| 你的场景 | 推荐产品 | 改造量 |
|----------|----------|--------|
| 数据量不大，想要云原生弹性和 Serverless | **TDSQL-C** | **极小**，几乎零改造，100% 兼容 MySQL |
| 数据量大，需要水平扩展（分库分表） | **TDSQL MySQL 版（分布式）** | **中等**，需要设计分片键、改造部分 SQL |
| 不需要分片，只需要高可用 | **TDSQL MySQL 版（集中式/NoShard）** | **极小**，等同 MySQL 主从 |
| PB 级数据 + 多模态需求 | **TDSQL Boundless** | **较大**，新架构，需要全面评估 |

> **关键决策点**：如果你不需要分库分表，选 TDSQL-C 或集中式实例，改造量接近零。下面的注意事项主要针对 **TDSQL MySQL 版分布式实例**，这是改造量最大的场景。

## 2. 不兼容特性清单

以下 MySQL 特性在 TDSQL 分布式版中**不支持**，迁移前必须排查业务代码：

| 特性 | TDSQL 分布式版支持情况 | 改造建议 |
|------|----------------------|----------|
| **存储过程** | 不支持 | 迁移到应用层逻辑 |
| **触发器** | 不支持 | 迁移到应用层逻辑 |
| **自定义函数（UDF）** | 不支持 | 迁移到应用层 |
| **外键** | 不支持 | 从业务层保证数据一致性 |
| **视图** | 可创建但不推荐 | 改为应用层查询封装 |
| **全文索引** | 不支持 | 使用 Elasticsearch 等外部方案 |
| **临时表** | 不支持 CREATE TEMPORARY TABLE | 改为普通表或应用层处理 |
| **事件（EVENT）** | 不支持 | 使用外部定时任务（如 cron） |
| **表空间** | 不支持 | 无需关注 |
| **游标** | 不支持 | 迁移到应用层 |
| **复合语句（BEGIN...END）** | 不支持 | 拆分为单条 SQL |

> **TDSQL-C 没有这些限制**，如果你的业务重度使用存储过程/触发器，优先考虑 TDSQL-C。

## 3. 分片键设计（最重要的改造点）

分片键（shardkey）是 TDSQL 分布式版的核心概念，**决定了数据如何分布到不同节点**。这是 MySQL 迁移到 TDSQL 分布式版最大的改造点。

### 建表语法变化

```sql
-- MySQL 原始建表
CREATE TABLE orders (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  amount DECIMAL(10,2),
  created_at DATETIME
);

-- TDSQL 分布式版建表（末尾加 shardkey）
CREATE TABLE orders (
  id BIGINT AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  amount DECIMAL(10,2),
  created_at DATETIME,
  PRIMARY KEY (id, user_id)  -- 主键必须包含 shardkey
) shardkey=user_id;
```

### 分片键规则（必须遵守）

| 规则 | 说明 |
|------|------|
| 主键和所有唯一索引**必须包含** shardkey | 违反则建表失败 |
| shardkey 值**不能 UPDATE** | 否则报错 |
| shardkey 不能用中文值 | Proxy 不转换字符集，会路由到错误分区 |
| 支持的类型 | int/bigint/smallint/char/varchar/decimal/float |
| char/varchar 建议长度 ≤ 255 | 超过可能影响性能 |
| `shardkey=xxx` 放在 CREATE TABLE 最后 | 语法要求 |

### 如何选择 shardkey

| 原则 | 说明 |
|------|------|
| **选查询最频繁的等值条件字段** | 这样查询能精准路由到单个节点，避免全节点扫描 |
| **数据分布均匀** | 避免选取值很少的字段（如 status），否则会数据倾斜 |
| **优先用 hash 分片** | range/list 容易导致数据倾斜 |
| **考虑 JOIN 场景** | 两张表用相同 shardkey 值的行会落在同一节点，JOIN 效率高（组拆分） |

### 三种表类型选择

| 表类型 | 适用场景 | 建表方式 |
|--------|----------|----------|
| **分片表** | 数据量大、需要水平扩展 | `shardkey=字段名` |
| **广播表** | 配置表、字典表（数据量小、读多写少） | `shardkey=noshardkey_allset` |
| **单片表** | 数据量小且不需要 JOIN 分片表 | 不加 shardkey（默认存第一个节点） |

## 4. SQL 语法差异

### DML 限制

| MySQL 写法 | TDSQL 分布式版 | 改造方式 |
|------------|---------------|----------|
| `UPDATE t SET ... `（无 WHERE） | 不支持 | 必须加 WHERE 条件 |
| `DELETE FROM t`（无 WHERE） | 不支持 | 必须加 WHERE 条件 |
| `LOAD DATA INFILE ...` | 不支持 | 使用 DTS 或 INSERT 批量导入 |
| `INSERT INTO ... SELECT ...` | 版本 ≥ 1.14.4 支持 | 确认版本 |
| `SET @var=1; SELECT @var` | 不支持变量操作 | 在应用层处理 |
| `SELECT ... FOR UPDATE`（跨分片） | 支持但性能较差 | 尽量带 shardkey 条件 |
| `CREATE TABLE ... SELECT` | 不支持 | 先建表再 INSERT |

### 查询注意

| 场景 | 影响 | 建议 |
|------|------|------|
| 查询不带 shardkey | 路由到**所有节点**，性能差 | 尽量带 shardkey 条件 |
| 跨节点 JOIN | 性能低于单机 JOIN | 用组拆分（相同 shardkey）或广播表优化 |
| ORDER BY / GROUP BY | 跨节点需要 Proxy 汇总排序 | 大结果集性能较差，考虑应用层处理 |
| 子查询 | 支持，但跨节点性能一般 | 优先用 JOIN 替代子查询 |

### 管理 SQL 限制

以下需要用**透传语法**（指定发到某个分片）：

```sql
-- 透传到所有分片
/*sets:allsets*/ ANALYZE TABLE t;

-- 透传到指定分片
/*sets:set_1*/ SHOW PROCESSLIST;
```

## 5. 事务行为差异

| 维度 | MySQL | TDSQL 分布式版 |
|------|-------|---------------|
| 事务协议 | 单机事务 | 2PC 分布式事务（对客户端透明） |
| 隔离级别 | 默认 REPEATABLE READ | 支持 RC / RR / SERIALIZABLE |
| DDL 事务 | 不支持 | 不支持（与 MySQL 一致） |
| XA 事务 | 支持 | 支持，且提供 `select gtid()` 查询事务状态 |
| 事务异常恢复 | 自动 | 异常提交后 30 秒内由后台自动提交/回滚 |

**注意**：分布式事务的 2PC 会比单机事务有额外开销（约 10-20% 的延迟增加），但对客户端完全透明，代码无需改动。

## 6. 自增 ID 行为变化

| 维度 | MySQL | TDSQL 分布式版 |
|------|-------|---------------|
| 全局唯一 | 单实例内唯一 | **全局唯一**（跨分片） |
| 连续递增 | 保证连续 | **不保证连续**（分片缓存机制，默认 batch=100） |
| 自增列要求 | 无特殊限制 | 必须是主键或唯一索引的一部分 |

**影响**：如果业务依赖自增 ID 的连续性（如订单号），需要改为应用层生成（如 Snowflake），或设置 `tdsql_auto_increment_batch_size=1`（性能会下降）。

## 7. 迁移工具与流程

### 推荐工具

| 工具 | 说明 |
|------|------|
| **DTS（数据传输服务）** | 腾讯云官方，支持全量 + 增量迁移，推荐 |
| **DBbridge** | 异构数据库迁移，支持 Oracle/MySQL → TDSQL，自动转换 DDL/DML |
| **mysqldump + 导入** | 简单场景，手动操作 |

### DTS 迁移流程

```
1. 预检查
   ├── 源端表数量 ≤ 5000
   ├── 所有表使用 InnoDB 引擎
   ├── 无 Geometry 类型字段
   └── Binlog 格式为 ROW（非 STATEMENT）
       ↓
2. 全量迁移
   ├── DTS 无锁读取源端数据
   ├── 写入目标端
   └── 期间不影响源端业务
       ↓
3. 增量同步
   ├── 基于 Binlog 实时同步增量数据
   ├── 期间不要修改主键/分片键
   └── 不要在目标端双写
       ↓
4. 数据校验
   ├── DTS 自动对比数据一致性
   └── 使用 __tencentdb__ 库记录对比信息
       ↓
5. 业务切换
   ├── 低峰期停止源端写入
   ├── 确认增量同步追平
   └── 切换业务连接到 TDSQL
```

### 迁移期间禁止操作

- 不要修改主键或分片键
- 不要在目标端双写
- 不要清除源端 Binlog
- 不要删除 `__tencentdb__` 库
- 不要新增分片或调整分片规格

## 8. 性能调优要点

| 要点 | 说明 |
|------|------|
| 查询带 shardkey | 避免全节点扫描，性能差距可达 10-100 倍 |
| 关联表用组拆分 | 两张频繁 JOIN 的表用相同 shardkey，数据落在同一节点 |
| 小表用广播表 | 配置表、字典表设为广播表，JOIN 时等同单机性能 |
| 避免大事务 | 分布式事务有额外开销，拆分为小事务 |
| 避免跨节点大排序 | 大量 ORDER BY 数据需要 Proxy 汇总，考虑应用层排序 |
| 读写分离 | 读请求路由到从节点，降低主节点压力 |
| 单分片数据量 | 建议不超过 2000 万行 / 2TB |
| 字符集 | 统一使用 utf8mb4，建表时显式指定 |
| 事务隔离级别 | 建议使用 Read Committed（RC），性能优于 RR |

## 9. 迁移检查清单

### 迁移前评估

- [ ] 确认选择哪个 TDSQL 产品线（TDSQL-C / 分布式 / 集中式 / Boundless）
- [ ] 排查业务代码中是否使用存储过程、触发器、外键、UDF
- [ ] 排查是否使用临时表、全文索引、事件
- [ ] 排查是否有 `LOAD DATA`、不带 WHERE 的 UPDATE/DELETE
- [ ] 排查是否依赖自增 ID 的连续性
- [ ] 排查是否使用 SQL 变量（`@var`）

### 分布式版特有

- [ ] 为每张需要分片的表设计 shardkey
- [ ] 确认主键和所有唯一索引包含 shardkey
- [ ] 规划表类型：分片表 / 广播表 / 单片表
- [ ] 评估关联查询，规划组拆分策略
- [ ] 确认查询 SQL 都尽量带 shardkey 条件

### 迁移执行

- [ ] 源端 Binlog 格式设为 ROW
- [ ] 源端表数量不超过 5000
- [ ] 源端都用 InnoDB 引擎
- [ ] 创建 `__tencentdb__` 库并授权
- [ ] 在低峰期执行全量迁移
- [ ] 增量同步追平后在低峰期切换

### 迁移后验证

- [ ] 数据一致性校验通过
- [ ] 核心业务 SQL 功能测试通过
- [ ] 性能基准测试与 MySQL 对比无明显退化
- [ ] 分布式事务场景测试通过
- [ ] 故障切换演练通过
